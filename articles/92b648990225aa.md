## セキュリティ管理フレームワーク

フレームワークとは、構造化されたガイドラインやルールの枠組みのこと。
セキュリティ管理をするときに、とりあえずやっておけば上手くいくという、実績のあるお手本のようなもの。
フレームワークには以下の通りさまざまな種類がある。

### ・ISO/IEC 27000シリーズ

![](https://storage.googleapis.com/zenn-user-upload/daf09406ed3b-20260118.png)

ISO国際標準機構とIEC国際電気標準会議が共同で作成している規定。世界中で同じ品質のものを作るための基準を定めている。　

#### ISMS(情報セキュリティマネジメントシステム)

セキュリティを管理する上で手順と方針を整理したもの。一般的にISMSという場合、ISO/IEC27001を指すことが多い。
覚えておくとよいのは、ISO/IEC27001、27002の2つである。

| ISO/IEC項目 | 内容 |
| ---- | ---- |
| 27001 | ISMSの要求事項を定めた国際規格。認証を取得するための基準|
| 27002 | 27001の要求事項を満たすための実践方法。ベストプラクティスをまとめたもの。 |
| 27004 | ISMSの有効性を評価するための「測定」と「報告」に関する指針 |
| 27005 | 情報セキュリティにおけるリスクマネジメントのプロセスに特化した指針 |
| 27799 | 医療情報を扱う組織が、ISO/IEC 27002（管理策のベストプラクティス）をどのように適用すべきかを示した業界特化型の指針 |

国際規格である「ISO/IEC 27001」に適合していることを第三者機関（認証機関）に評価してもらい、適合していること証明してもらうことが可能である。発行された認証文書を外部組織に示すことで、自身の組織のセキュリティ管理が適切に実施できていることを証明することができる。

### ・NIST SP 800-37
米国国立標準技術研究所（National Institute of Standards and Technology）が定めた情報システム及び組織のためのリスクマネジメントフレームワークである。
このフレームワークはアメリカの連邦政府システムだけではなく、すべての企業を対象としている。
個人データ（プライバシー）の保護に関しては、セキュリティと同等あるいはそれ以上に重要視されてきている。
組織のリスクをコントロールすることを主な目的としており、ISO/IEC 27001と比較するとより実践的な内容となっている。両社は対立関係ではなく補完関係である。両社の考えを企業に合わせて適切に取り入れると効果的である。

:::message
厳密には違うかもしれないが、ISO/IEC27001は建築基準法、NIST SP 900-37は建築手順書というイメージ。
:::

#### NIST リスクマネジメントフレームワーク（RMF）

![](https://storage.googleapis.com/zenn-user-upload/0a616d1e6e8c-20260118.png)

| ステップ | ステップ名 | 主な内容 |
| ---- | ---- | ---- |
| 1 | 準備 (PREPARE) | リスクマネジメント戦略の策定や、複数のシステムで共有可能な共通管理策の特定などが含まれる。 |
| 2 | 分類 (CATEGORIZE) | 対象となる情報システムと、それが扱う情報を分類する。情報の種類に基づき、機密性・完全性・可用性のそれぞれの影響度（低・中・高）を決定し、システムのセキュリティカテゴリを確定する。 |
| 3 | 選択 (SELECT) | システムのセキュリティカテゴリに基づき、適切な初期セキュリティ管理策セット（ベースライン）を選択する。さらに、組織の特定の要件や環境に合わせて、その管理策の内容を調整する。 |
| 4 | 実装 (IMPLEMENT) | 選択・調整されたセキュリティ管理策を、対象の情報システムおよびその運用環境に実際に実装・適用する。 |
| 5 | 評価 (ASSESS) | 実装されたセキュリティ管理策が正しく実装され、意図した通りに動作し、定められたセキュリティ要件を満たしているかを評価（アセスメント）する。 |
| 6 | 認可 (AUTHORIZE) | 認可責任者が、アセスメント結果に基づきリスクを評価する。組織の運用に対するリスクが受容可能であると判断された場合、システムの運用を公式に認可する。 |
| 7 | 監視 (MONITOR) | 運用中のシステムにおけるセキュリティ管理策の有効性を継続的に監視・評価する。環境の変化や新たな脅威に対応し、継続的な運用認可の判断を行う。また、システム廃止時には必要なセキュリティ措置を講じる。 |

フレームワーク全体のベースとなる準備（PREPARE）は、組織レベルとシステムレベルの2つの観点がある。

- 組織レベル：組織全体のセキュリティ、プライバシー保護を評価、管理する。リスクマネジメントや、サプライチェーンリスクマネジメントなど
- システムレベル：特定のシステムのリスクを評価、管理する。資産管理、リスクアセスメントなど。

#### ・FedRAMP

![](https://storage.googleapis.com/zenn-user-upload/7fcac430d90d-20260118.png)

クラウドサービスを採用する際の、セキュリティ評価、認証に関する認定制度である。アメリカの公的機関がクラウド導入する場合、そのサービスはFedRAMPの認定受けなければならない。
アメリカ全体で安全なクラウドサービスの導入を促進することを目的としている。AWS、AZure、GCPなのでクラウドサービスの普及の促進をした。

NIST 800-37がFedRAMPの土台となっており、クラウドサービスプロバイダはFISMA（連邦情報セキュリティマネジメント法）に準拠している必要がある。

FISMAはアメリカの法律であり、FedRAMPはその法律をクラウドサービスに適用するためのプログラムである。

## まとめ

- ISO/IEC 27000シリーズは組織全体で情報セキュリティを体系的に管理・運用するための仕組み
- NIST SP 800-37とは対立関係ではなく補完関係である
- FedRAMPはクラウドセキュリティに関する評価、認定制度
