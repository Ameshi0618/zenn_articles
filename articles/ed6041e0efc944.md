### 記事の対象者

## JMLプロセス

従業員のライフサイクルはJMLプロセスに基づいている。
セキュリティ担当者としては、入社（Joiner）、異動（Mover）、離職（Leaver）の3つを意識すべきである。

| フェーズ | 主な管理策 | ポイント |
| ---- | ---- | ---- |
| 入社 | バックグラウンドの調査、機密保持契約（NDA） | 職務の重要度に応じた調査を行う |
| 異動 | 職務分離、権限の変更 | 権限の肥大化を阻止する |
| 離職 | アカウントの無効化 | 敵対的な解雇に注意 |

![](https://storage.googleapis.com/zenn-user-upload/31c755abe3df-20260205.png)

### 入社

- バックグランドチェック: 全員一律ではなく、扱うデータの機密性に応じて調査度合いを調整する。
- 入社時に機密保持契約（NDA）: 入社時に署名をさせるが、退職後も有効であることを明確にすること。

:::message
入社する際は、新しい社員としてアカウントを登録する必要があるが、それを管理するシステムをIAM（Identity and Access Management）システムという。
:::

### 異動

- 最小権限の原則: 新しい部署に異動した際は、新しい権限を与えるだけでなく、古いお部署の権限を削除することが重要である。これを忘れてしまうと一人の人間が強大な権限を持つ「権限の肥大化」を招くことになる。

### 解雇・退職

- アカウント管理: 退職者のアカウントは削除ではなく無効化が推奨されることが多い。これは、監査証跡の維持のためである。
- 退職面談: 退職者の不満などを聞くことで組織の改善点を洗い出す。また、セキュリティ上の義務を再確認する最後の機会です。

:::message
退職者の情報持ち出しなどが退職後に判明した際に、情報が残っていないと困るからである
:::

#### 敵対的な解雇

従業員を解雇する場合、解雇を告げる前に権限を剥奪するのが良い。解雇通告後に逆上してデータなどを破壊されるリスクを防ぐためである。

## 競業避止義務

雇用期間中、及び退職後も一定期間は同業他社で働いたり、競合するビジネスを始めたりしないという義務である。

:::message
会社員であれば、自社の就業規則に競業避止義務についての記載があるだろう
:::

## まとめ

- 入社時は機密保持契約の署名をさせる
- 異動の際は権限を適切に変更し権限の肥大化を防止する
- 退職時はアカウントを削除ではなく無効化にする
