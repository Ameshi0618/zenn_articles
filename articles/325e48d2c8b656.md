## 脅威モデリング

システム設計をする上で、どのような攻撃者が、どのような経路で攻撃してくる可能性があるかを
特定し、対策を検討するプロセスである。
設計、実装後に脆弱性を発見し対応するよりも、事前に脆弱性を特定し対策する方がコストを抑えられる。

:::message
全ての脅威を100％対策することは不可能なため、資産の価値や、脅威の発生確率などを考慮し、優先度を決める必要がある。
:::

![](https://storage.googleapis.com/zenn-user-upload/b94f15266979-20260217.png)

### STRIDE

6つの脅威カテゴリの頭文字をとったものである。脅威を特定するのに利用できる。

| 項目 | 脅威 | 損なう特性 | 概要 |
| ---- | ---- | ---- | ---- |
| S | なりすまし(Spoofing) | 真正性 | 正当なユーザーを装うこと |
| T | 改ざん(Tampering) | 完全性 | データを不正に書き換えること |
| R | 否認(Repudiation) | 否認防止 | 事実として認めさせる証拠がないこと |
| I | 情報漏洩(Information Disclosure) | 機密性 | 権限のない者が情報を見れてしまうこと |
| D | サービス拒否(Denial of Service) | 可用性 | 正当なユーザーがサービスを使用できなくなること |
| E | 権限昇格(Elevation of Privilege) | 認可 | 一般権限でログインし管理者権限を奪うこと |

:::message
どういう対策がその脅威カテゴリの対策になるのかを意識するとよい。
例えば、デジタル署名は改ざんと否認の防止に役立つ。
:::

### DREAD

脅威が組織にとってどれほど深刻なリスクなのかを数値化して評価するためのモデル。
STRIDEで脅威を特定、分類した後はDREADで評価する必要がある。

| 項目 | 評価項目 | 概要 |
| ---- | ---- | ---- |
| D | 潜在的損害（Damage） | 攻撃が成功した場合の被害の大きさ |
| R | 再現性（Reproducibility） | 攻撃が簡単に繰り返せるか |
| E | 攻撃可能性（Exploitability） | 攻撃が簡単に実行できるか |
| A | ユーザへの影響（Affected Users） | 被害を受けるユーザーの割合 |
| D | 発見可能性（Discoverability） | 脆弱性がどれだけ見つけやすいか |

#### リスク値の算出方法

一般的には、各項目の平均値を出してリスクスコアとする。
$$Risk = \frac{D + R + E + A + D}{5}$$

:::message
各項目に対して1~10で評価するのが一般的である
:::

### PASTA

ビジネス重視の脅威モデリングの手法であり、セキュリティをビジネスの目標やガバナンスと整合させるための戦略的なフレームワークである。

| ステップ | 項目 | 概要 |
| ---- | ---- | ---- |
| 1 | 目的の定義 | ビジネス上の目的、コンプライアンス要件等を定義 |
| 2 | 技術的範囲の定義 | システムの境界、コンポーネント、依存関係を明確にする |
| 3 | アプリケーションの分解 | データフローを作成し、データの流れを可視化する |
| 4 | 脅威分析 | 具体的な脅威を特定する |
| 5 | 脆弱性分析 | 脆弱性を特定し、想定される悪用方法を洗い出す |
| 6 | 攻撃モデリング | 攻撃者が脆弱性どのように攻撃するかをシミュレーションする |
| 7 | リスク分析 | ビジネスへの最終的なリスクを評価し対策を決定する |

![](https://storage.googleapis.com/zenn-user-upload/1986cd37485c-20260217.png)

## まとめ

- 脅威モデリングは設計実装の事前に実施することが重要
- STRIDEで脅威を分類し、DREADで評価する
- PASTAはビジネス重視の脅威モデリング手法
