## 事業継続計画(BCP)

事業継続計画を立てる目的は、災害やシステム障害が発生した際に、「業務を中断させない」、あるいは「許容される時間内に業務を再開させる」ことにある。これは情報セキュリティの3要素のうちの可用性を支えるものである。

:::message
事業継続計画がしっかりできていない場合、復旧のめどが立たず、ステークホルダーを安心させることができません。
:::

## BCP策定の流れ

### ステップ1: 方針決定

組織全体の構造を把握し、組織全体を対象とするのか、特定のプロジェクトやサービスを対象とするのかを定義する。
重要なのは「経営陣の承認と関与」を得ることであり、予算や人員の確保ができなければ、BCPの意味がありません。

### ステップ2: ビジネスインパクト分析（BIA）

各業務が停止した際の影響を分析し、優先順位を決定する。
事業活動において、どの業務が止まると一番影響が大きいのかを定量的に算出する。
BCP策定の手順において最重要プロセスである。

::: message
企業において最重要な業務というのは異なるため、自社のビジネスモデルに詳しい人に聞いてしまうのが早い。
:::

#### 稼働率

- MTBF（平均故障間隔）: 復旧してから故障するまでの平均時間
- MTTR（平均復旧時間）: 故障してから復旧するまでの平均時間

$$\text{稼働率 (\%)} = \left( \frac{MTBF}{MTBF + MTTR} \right) \times 100$$

::: message
稼働率が100%に近いのはいいことだが、その分コストがかかる。ビジネスの損失コストと、対策コストの均衡するポイントが理想の稼働率となる。つまり投資対効果を最も高くなるようにコストをかけるべきである。
:::

::: message
理想の稼働率は業界によっても異なる。社内の勤怠管理システムなどであれば99.0%で十分だと思われるが、医療分野では100%に近い値が求められるだろう。
:::

#### RASIS

システムの品質を評価する指標としてRASISというものがある。

| 頭文字 | 項目 | 意味 | 指標 |
| ---- | ---- | ---- | ---- |
| R | 信頼性（Reliability） | 故障のしにくさ | MTBF |
| A | 可用性（Availability） | サービスを長く利用できること | 稼働率 |
| S | 保守性（Serviceability） | 復旧のしやすさ | MTTR |
| I | 完全性（Integrity） | データが正しく改ざんされてないこと | - |
| S | 安全性（Security） | 不正アクセスや情報漏洩を防げること | - |

#### 最大許容停止時間（MTD）

最大許容停止時間を超えてしまうと、組織として再起不能になる限界点のことである。

- 目標復旧時点（RPO）: 過去のどの時点までデータを戻すかの目標値。言い換えればどの程度までデータ損失を許容できるかということである。
- 目標復旧時間（RTO）: システムが再び稼働するまでの時間
- 業務復旧時間（WRT）: システム復旧後、通常業務ができる状態になるまでの時間
- 最小動作要件（MOR）: システム復旧に必要な最低限の環境

::: message
RPOはバックアップの取得頻度に直結する
:::

以下の条件を満たす必要がある
$RTO + WRT \le MTD$

### ステップ3: リスク評価（RA）

BIAが業務が止まった場合の影響度合いを調査するのに対し、リスク評価では、何が原因で業務が止まるか、という脅威を分析する。
BIAの結果とリスク評価の結果を掛け合わせ、対策の優先順位を決定する。

仮に、業務が止まった場合の影響度は小さいが、脅威が多く発生頻度が高いのであれば優先度を上げたほうがよい。

::: message
サーバーの冗長化などに目が向きがちだが、人の冗長化も忘れてはならない。属人化が発生している業務は、組織にとって「単一障害点」そのものである
:::

### ステップ4: 戦略策定と計画の文書化

これまでの調査、分析の結果をもとにマニュアル化する。
インシデント発生時の初動対応、復旧作業の手順など、具体的に記載する。

## まとめ

- BCP策定において、各業務停止時の影響度と脅威を分析し優先順位を決める
- 最大許容停止時間以内にシステムは復旧しなければならない
